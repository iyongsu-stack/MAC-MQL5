## 🔺 피라미딩 (Pyramiding) 기법 정리

피라미딩은 **기존 포지션이 수익 중일 때 추가 진입하여 수익을 극대화**하는 기법입니다. 특히 현재 프로젝트의 **트렌드 추종 + TrailingStop** 구조와 매우 잘 맞습니다.

---

### 1. 피라미딩의 핵심 원리

```
가격 ↑
  │         ┌──── 3차 추가 (최소 랏)
  │     ┌───┤
  │ ┌───┤   └──── TrailingStop ━━━━━━━━━━┐
  │ │   └──── 2차 추가 (소 랏)            │ 
  │ │                                     │ 전체 청산
  ├─┤──────── 1차 진입 (기본 랏)          │
  │ │                                     ▼
──┴─┴───────────────────────────────────────── 시간 →
```

**핵심**: 추세가 진행될수록 포지션을 **점진적으로 증가**시키되, 각 추가 진입의 랏 사이즈는 **감소**시킴

---

### 2. 피라미딩 유형

| 유형 | 랏 배분 예시 | 특징 | 리스크 |
|:---|:---|:---|:---|
| **정피라미드** (Standard) | 1.0 → 0.5 → 0.25 | 가장 보수적, 평균 단가 유리 | ⭐ 낮음 |
| **균등 피라미드** (Equal) | 0.5 → 0.5 → 0.5 | 구현 간단 | 중간 |
| **역피라미드** (Inverted) | 0.25 → 0.5 → 1.0 | 최대 수익 잠재력 | ⚠️ 매우 높음 |

> 💡 **권장**: 정피라미드 (Standard). 추세 초기에 가장 크게 진입하고, 추가 진입 시마다 줄여감

---

### 3. 피라미딩 조건 설계 (BSP 프레임워크 맥락)

현재 시스템 구조에 맞는 피라미딩 조건:

```text
[1차 진입] AI 신호 발생 (BSP>1.5 + TV>1.2× + AI 확률 > 0.6)
    → 기본 랏 진입, TrailingStop 활성화

[2차 추가 조건]
    ✅ 1차 포지션이 ATR × 0.5 이상 수익 중
    ✅ BSP Scale이 여전히 > 1.5 (추세 지속 확인)
    ✅ AI 재확인 신호 (선택적)
    → MoneyManageVx가 가용 증거금(Free Margin)과 최신 ATR을 기반으로 동적 랏 사이즈(예: 1차 대비 50%) 계산 후 진입

[3차 추가 조건]
    ✅ 전체 (그룹) 포지션이 총 ATR × 1.0 이상 누적 수익 중
    ✅ BSP Scale > 2.0 (강한 추세 확인)
    → MoneyManageVx가 가용 증거금 기반으로 동적 랏 사이즈(예: 1차 대비 25%) 계산 후 진입

[공통 관리 및 청산 (Hedging 계좌 반영)]
    → MT5 Hedging 계좌 특성상 다중 진입은 각각 별도의 티켓 번호로 생성됨.
    → TrailingStopVx 모듈은 동일한 '매직 넘버(Magic Number)'를 가진 그룹 포지션의 통합 누적 PnL을 모니터링하여 조건 충족 시 일괄 청산.
```

---

### 4. 피라미딩의 수학적 효과

**기본 시나리오** (피라미딩 없이):
- 1.0 랏 × 100pt 수익 = **100pt**

**피라미딩 시나리오** (정피라미드):
| 진입 | 랏 | 진입가 대비 수익 | 기여 수익 |
|:---:|:---:|:---:|:---:|
| 1차 | 1.00 | +100pt | 100pt |
| 2차 (+50pt 시점) | 0.50 | +50pt | 25pt |
| 3차 (+80pt 시점) | 0.25 | +20pt | 5pt |
| **합계** | **1.75** | | **130pt** |

> 📈 같은 추세에서 **+30% 추가 수익** 확보 (리스크는 TrailingStop으로 통제)

---

### 5. 리스크 관리 핵심 규칙

| 규칙 | 설명 |
|:---|:---|
| **최대 추가 횟수** | 2~3회로 제한 |
| **총 노출 한도** | 전체 포지션 합이 최대 리스크의 2배 미달 (Free Margin 기반 `MoneyManageVx` 랏 제어) |
| **추가 진입 조건** | 매직넘버가 동일한 그룹 포지션 전체가 **반드시 통합 수익 중**이어야 함 |
| **통합 손절선 상향 (B/E)** | 추가 진입 즉시, 전체 그룹 포지션의 **평균 단가(Average Open Price) + 총 마찰비용(30pt × n)** 이상으로 모든 포지션의 SL을 일괄 이동시켜 하락 반전 시 원금 손실을 완벽 방어 |
| **일괄 청산** | TrailingStop 터치 시 그룹 내 **전체 포지션 동시 청산** |

---

### 6. 현재 시스템과의 시너지

현재 구조가 피라미딩에 **이상적인 이유**:

1. **TrailingStop 전담 청산** → 추세가 끝날 때까지 보유, 피라미딩과 궁합 최적
2. **BSP Scale 실시간 모니터링** → 추세 강도 기반 추가 진입 판단 가능
3. **AI 확률 출력** → 추가 진입 시점에서도 AI 재확인 가능
4. **ATR 기반 동적 간격** → 추가 진입 간격을 변동성에 맞게 자동 조절

---

### ⚠️ 주의사항 (Strategic Advisor)

1. **Walk-Forward에서 피라미딩 포함 검증 필수** — 피라미딩 없는 승률과 피라미딩 포함 총 수익을 별도 비교
2. **시스템 모듈 통합 필수** — `OpenCloseVx`(진입 판단), `MoneyManageVx`(동적 랏+증거금 계산), `TrailingStopVx`(그룹 포지션 판별 및 평균단가 B/E 업데이트) 간 유기적 통신 구현이 핵심
3. **Friction Cost 누적** — 추가 진입마다 30pt × 진입 횟수만큼 마찰 비용이 가중되므로 B/E 이동 시 반드시 고려
4. **먼저 피라미딩 없이 기본 전략을 완성한 후** 피라미딩을 별도 모듈로 추가하는 것이 안전

---

## 🎯 피라미딩 진입 시점 AI 학습

피라미딩 진입 시점도 **AI가 학습할 수 있습니다.** 핵심은 라벨링 방식을 어떻게 설계하느냐입니다.

### 접근법: 멀티 라벨 학습 (Multi-Label)

현재 시스템은 **단일 라벨**(진입 여부 1개)이지만, 피라미딩을 학습시키려면 **별도의 라벨**이 필요합니다:

```
[기존] Model_Entry:   "지금 진입할까?" → label_long (0 or 1)

[확장] Model_AddOn:   "기존 포지션에 추가할까?" → label_addon (0 or 1)
```

### 구체적 설계: 2-모델 구조

```
                    ┌──────────────────────┐
  M1 봉 데이터 ──→ │  Model_Entry (기존)  │ → 1차 진입 신호
                    │  480 피처            │
                    └──────────────────────┘

                    ┌──────────────────────┐
  M1 봉 데이터 ──→ │  Model_AddOn (신규)  │ → 2차/3차 추가 신호
  + 포지션 상태 ──→ │  480 + α 피처        │
                    └──────────────────────┘
```

### Model_AddOn에 추가되는 피처 (α)

| 피처 | 설명 |
|:---|:---|
| `unrealized_pnl_atr` | 현재 미실현 수익 / ATR (정규화) |
| `bars_since_entry` | 진입 후 경과 봉 수 |
| `bsp_scale_delta` | 진입 시점 대비 BSP Scale 변화 |
| `price_vs_entry` | (현재가 - 진입가) / ATR |
| `trend_acceleration` | BSP 기울기의 변화율 (추세 가속 중인가) |
| `addon_count` | 이미 추가한 횟수 (0, 1, 2) |

### 라벨링 방법 (label_addon)

기존 **Barrier 라벨링을 확장**합니다:

```python
# 1차 진입이 발생한 이후 시점에서만 라벨 생성
# 조건: 이미 1차 진입이 수익 중 (unrealized_pnl > ATR × 0.3)

# label_addon = 1 (추가 진입 성공):
#   → 추가 진입 후에도 TP 배리어에 먼저 도달
#   → 추가 진입분 자체가 수익

# label_addon = 0 (추가 진입 실패):
#   → 추가 진입 후 SL 배리어에 먼저 도달
#   → 추가 진입이 오히려 손실 확대
```

### ⚡ 더 간단한 대안: 단일 모델 + 확률 임계값

2개 모델이 복잡하다면, **기존 1개 모델의 확률 출력**을 활용할 수도 있습니다:

```
AI 확률 > 0.6  →  1차 진입
AI 확률 > 0.7  →  2차 추가 허용 (기존 포지션 수익 중일 때)
AI 확률 > 0.8  →  3차 추가 허용
```

이 방식은 **추가 학습 없이** 기존 모델만으로 가능하고, 구현도 훨씬 간단합니다.

### 📌 권장 구현 순서

| 단계 | 내용 | 복잡도 |
|:---:|:---|:---:|
| **1단계** | 피라미딩 없이 기본 전략 완성 + Walk-Forward 검증 | ⭐ |
| **2단계** | 확률 임계값 기반 피라미딩 (규칙 기반) | ⭐⭐ |
| **3단계** | Model_AddOn 별도 학습 (AI 학습 기반) | ⭐⭐⭐ |

> 💡 **1단계를 먼저 완성**하고, 실전 성능이 확인된 후 2→3단계로 확장하는 것이 과적합 방지에도 안전합니다.

### 🛠 데이터 파이프라인 업데이트 요구사항 (AI 학습 확장 시)
Model_AddOn(멀티 라벨) 방식 도입을 확정할 경우, 현재 구축된 **4계층 Data Lake 파이프라인의 수정이 필수**입니다.
- **`build_labels_barrier.py` 스크립트 수정**: 기존 라벨(Model_Entry) 이외에 추가 진입용 라벨(label_addon)을 생성하는 로직 추가 개발 필요.
- **신규 로직**: "1차 진입이 수익 중일 때, 현재 시점에서 추가 진입했을 경우 이후 가격 궤적이 TP 배리어를 먼저 치는가 SL 배리어를 먼저 치는가?"를 평가하여 `1` 또는 `0` 라벨링.
