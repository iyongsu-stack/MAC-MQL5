# 1.결측치(NaN/INF) 방어

COnnxPredictor.mqh
에는 features[] 배열 내 NaN/INF를 체크하는 로직이 전혀 없습니다. 현재는 사이즈 불일치만 검증합니다 (Line 157-165):

cpp
if(inputSize != m_featureCount)  // ← 이것만 검증
이것이 왜 매우 위험한가 — 퀀트 시스템 관점
ONNX Runtime 크래시: NaN이 입력되면 ONNX Runtime은 크래시하거나 전혀 의미없는 확률값(예: 0.999)을 반환할 수 있음. 잘못된 확률 → 잘못된 진입 → 실질 손실
MQL5 지표 NaN 발생 시나리오:
바 데이터 부족: 시작 후 첫 240봉 동안 zscore240 계산 불가
매크로 CSV 로딩 지연: GOLD_zscore_240 등이 아직 로드 안 됨
커스텀 지표 계산 오류: iCustom()이 EMPTY_VALUE를 반환
주말/공휴일 후 첫 봉: 일부 상위 TF 지표가 아직 완성되지 않음
조용한 실패(Silent Failure): OnnxRun이 성공하더라도, NaN이 섞인 피처로 도출된 확률값은 통계적으로 무의미. 에러 로그도 남지 않아 디버깅 불가.

Reject 후 흐름
M1 봉 생성 → 피처 수집 (80개) → NaN 검사
                                    │
                         ┌──────────┴──────────┐
                         │                     │
                    NaN 발견               NaN 없음
                    Predict()=-1.0         Predict()=정상 확률
                         │                     │
                         ▼                     ▼
                  prob < 0.20 이므로      prob ≥ 0.20 ?
                  자연스럽게 진입 스킵         ├── Yes → 진입
                         │                    └── No  → 스킵
                         ▼
                  **다음 M1 봉까지 대기**
                  (1분 후 피처 재계산)
                         │
                         ▼
                  대부분 NaN 해소됨
                  (지표 warm-up 완료, 데이터 로드 완료)
핵심: 아무것도 안 합니다. 그냥 다음 봉을 기다립니다.
Predict()가 -1.0을 반환하면, EA의 진입 조건 prob ≥ 0.20에서 자연스럽게 탈락합니다. 별도의 재시도(Retry) 로직이 필요 없습니다.

왜냐면:

M1 기반 이벤트 루프: 매 1분마다 OnTick() → 새 봉 감지 → 피처 재계산 → Predict() 호출이 반복됨
NaN 원인은 대부분 일시적: 지표 warm-up(첫 240봉), CSV 로딩 지연, 주말 후 첫 봉 등 → 다음 봉에서 자연 해소
영구 NaN이면: 매 봉마다 계속 -1.0 → 계속 스킵 → 거래 안 함 = 안전. 이 경우 로그(Print로 어떤 피처가 NaN인지 기록)가 남아 운영자가 원인 추적 가능
실전에서 신경 써야 할 예외 케이스
상황	대응
EA 시작 후 첫 240분	지표 warm-up으로 NaN 다수 → 정상. 240봉 이후 자동 해소
매크로 CSV 미로드	CMacroLoader 초기화 실패 시 매크로 피처 전체 NaN → Init() 단계에서 감지/로그
연속 N봉 이상 NaN	이건 비정상. 예를 들어 500봉(≈8시간) 연속 NaN이면 Alert 발행 → 운영자 개입


#2. CMacroLoader의 '발표 시간' 지연 (Lag) 문제 인식

현재 CMacroLoader.mqh에는 발표 시간(예: 22:00)이 실제 데이터 시간(예: 21:59)보다 1분 늦게 로드되는 문제가 있습니다. 이로 인해 발표 직후 피처가 1분 지연되어 잘못된 신호가 생성될 수 있습니다.
